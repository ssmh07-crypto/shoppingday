# 운영 및 아키텍처 결정

이 문서는 Codespace나 AI 대화 세션이 초기화되어도 합의된 운영 방식을 잃지 않기 위한 기준 문서다. 이후 구현은 아래 결정을 우선 확인한다.

스마트스토어 및 네이버 커머스API 작업을 시작할 때는 [`naver-commerce-api.md`](naver-commerce-api.md)의 공식 문서 인덱스와 인증정보 관리 원칙을 먼저 확인한다.

## 공급사 API와 상품 이미지

- 친구도매 상품 API는 한국 시간 기준 하루 최대 5회만 호출한다.
- 최초 상품 가져오기와 관리자가 명시적으로 누르는 `공급사 최신 정보 갱신`에서만 API를 호출한다.
- 상품 상세·수정 화면을 열 때는 공급사 API를 호출하지 않고 PostgreSQL에 저장된 데이터를 사용한다.
- 이미 가져온 상품번호를 다시 가져오면 기존 DB 데이터를 반환하고 API 호출 횟수를 쓰지 않는다.
- 공급사 응답의 이미지 URL 문자열만 `supplier_products.original_images`에 저장한다.
- 관리 화면에서는 공급사 이미지 URL을 일반 `<img>`로 직접 표시한다. 이미지 파일을 R2나 Supabase Storage에 복사하지 않는다.
- Cloudflare R2는 향후 고객용 쇼핑몰에서 이미지의 장기 보존·최적화가 실제로 필요해질 때만 사용한다.
- Supabase는 인증과 PostgreSQL 용도이며 상품 이미지 저장소로 사용하지 않는다.

## 비용 원칙

- 단순 미리보기를 위한 이미지 중복 저장과 변환 비용을 만들지 않는다.
- 외부 API 호출은 자동 화면 조회와 분리하고 사용자가 비용을 알 수 있는 명시적 동작으로 둔다.
- 공급사 이미지 직접 로딩이 차단되거나 URL이 만료되는 것이 확인되기 전에는 별도 이미지 저장을 도입하지 않는다.

## 판매 상품 편집

- `supplier_products`는 원본 읽기 전용, `products`는 판매용 편집 마스터다.
- 상품 편집·저장·ready 전환·원본 이미지/옵션 초기화 중에는 외부 API를 호출하지 않는다.
- 판매 이미지 파일은 저장하지 않고 공급사 URL의 선택·대표·순서 메타데이터만 저장한다.
- 수동 저장과 낙관적 잠금을 사용하며 자동저장은 후속 검토한다.

## 네이버 스마트스토어 연동

- 운영 호출은 고정 공인 IPv4 VM의 제한형 HTTPS 중계 서버를 사용한다. Worker는 HMAC 서명 요청만 보내고 네이버 Client Secret과 Access Token은 중계 서버에만 둔다.
- 범용 전달 프록시는 만들지 않으며 중계 서버에 허용 API를 명시적으로 추가한다. 상세 운영 방식은 [`naver-commerce-relay.md`](naver-commerce-relay.md)를 따른다.
- 네이버 Client Secret과 Access Token은 서버에서만 사용하고 DB·Git·일반 로그·브라우저 응답에 저장하지 않는다.
- 카테고리 전체 동기화는 관리자가 누른 명시적 버튼에서만 네이버 API를 1회 호출한다.
- 카테고리 화면 조회와 상품 편집 중에는 저장된 `naver_commerce_categories`를 사용한다.
- 전체 카테고리 응답이 비어 있거나 형식이 잘못되면 기존 카테고리를 삭제하지 않는다.
- 상품 등록·수정·삭제는 커머스API 응답과 외부 상품번호를 보관하는 별도 채널 발행 모델을 추가한 후 활성화한다.

## 판매가와 실마진

- 등록 전 판매가는 공급가, 예상 수수료율, 판매자 부담 배송비, 포장·고정비, 구매자 배송비와 목표 실마진율로 역산한다.
- 등록 전 수수료는 설정 가능한 예상치이며 확정값으로 표현하지 않는다.
- 실제 이익은 주문 후 네이버 정산 API의 실제 수수료·할인·정산액과 배송비를 대조해 보정하는 구조로 확장한다.
